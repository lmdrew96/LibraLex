## High Priority Findings

### 1. ~~No app icons, web manifest, or OG metadata~~ â€” âś… SHIPPED
- **What:** There is no `public/` directory at all. No favicon, no app/touch icons, no `manifest.json`, no `metadataBase`, no Open Graph image, and `viewport` has no `themeColor`. `app/layout.tsx` metadata is just `title` + `description`.
- **Why it matters:** Production target is `libra.adhdesigns.dev`. On launch the browser tab shows a blank default icon, sharing the URL renders no card, "Add to Home Screen" has no icon, and mobile browser chrome won't pick up the brand color. For an app this visually polished, this is the one place it currently reads as unfinished.
- **Suggested action:** Add `app/icon.png` (+ `apple-icon.png`), `app/opengraph-image.png`, a `manifest.ts`/`manifest.json` with the Meadow/Twilight palette, `metadataBase: new URL("https://libra.adhdesigns.dev")`, and `themeColor` in `viewport`. The `pwa-setup` skill covers the manifest/installability half.
- **Effort estimate:** Half-day

### 2. ~~No error / not-found boundaries~~ â€” âś… SHIPPED
- **What:** No `app/error.tsx`, `app/global-error.tsx`, or `app/not-found.tsx`. Mutations are wrapped in try/catch with toasts (good), but a thrown Convex *query*, a render error, or an unknown route falls through to Next's default error screen.
- **Why it matters:** A single unhandled error drops the user out of the calm, branded experience into a raw stack-ish page â€” especially jarring for the ND-first audience this app targets.
- **Suggested action:** Add a branded `app/error.tsx` (reset button), `app/not-found.tsx`, and a minimal `app/global-error.tsx`, all using the existing `EmptyState` + `Button` components.
- **Effort estimate:** Quick fix

## Medium Priority Findings

### 4. README architecture section has drifted from the code
- **What:** `README.md` documents `reading/page.tsx` ("Currently Reading") â€” the actual route is `history/page.tsx` (Reading/Read tabs). The architecture map also predates the social/discovery work: it omits the `search`, `recs`, `friends`, `author`, `settings`, and `add/[code]` routes, the `api/discover|book-info|enrich` endpoints, and the `users`/`friendships`/`recommendations`/`dismissedBooks` tables. (The `convex/schema.ts` doc-comments, by contrast, are excellent and current.)
- **Why it matters:** The README is the first thing future-you or a collaborator reads; a stale map sends them to files that don't exist.
- **Suggested action:** Refresh the architecture block to match the current tree. The `docs-gen` skill can drive this.
- **Effort estimate:** Quick fix

### 5. Native `confirm()` dialogs break the themed experience (5 sites)
- **What:** `confirm()` is used for destructive/irreversible actions in `app/settings/page.tsx` (revoke MCP link, clear finish dates), `app/book/[id]/page.tsx` (delete), `app/friends/page.tsx` (remove friend), and `app/wishlist/page.tsx` (remove).
- **Why it matters:** Native dialogs are unstyled, can't carry the calm brand voice, and render inconsistently across mobile browsers â€” a sharp edge in an otherwise cohesive, ND-considered UI. The app already ships a Radix `Dialog`.
- **Suggested action:** Build one shared `<ConfirmDialog>` on the existing Radix primitive and replace the five `confirm()` calls.
- **Effort estimate:** Half-day

### 6. Recommendation + loan-period logic is triplicated with "keep in sync" comments
- **What:** Taste-subject tallying and candidate scoring exist in `lib/recommend.ts` (client), `convex/discover.ts` (Convex), and again in `convex/mcpData.ts` (`recommendInputsForUser`). The 21-day `LOAN_PERIOD_MS` is defined three times (`convex/books.ts`, `convex/mcpData.ts`, `lib/loans.ts`) â€” `mcpData.ts` even carries a `// Keep in sync` comment.
- **Why it matters:** The duplication is partly real (the Next/Convex runtime boundary can't always share modules), but "keep in sync by hand" is exactly how the in-app Discover row and the chat `recommend_books` results silently diverge over time. The loan constant has no excuse â€” it can be exported.
- **Suggested action:** Export `LOAN_PERIOD_MS` from one module and import it everywhere reachable. For the rec logic, extract the shared scoring/tally into a pure module both runtimes can import, or add a comment-linked test that asserts the two paths agree on a fixture.
- **Effort estimate:** Half-day

## Low Priority / Nice-to-Have

- **No tests on the riskiest pure logic.** `lib/loans.ts` (TZ-sensitive day-boundary math) and `lib/recommend.ts` (355 lines of scoring) are pure functions â€” ideal unit-test targets, and the date math is precisely the class of code that drifts by one across DST/zones. A tiny Vitest suite on these two would buy outsized confidence. *(Effort: half-day)*
- **Shelf filter chips lack `aria-pressed`.** The Settings genre chips correctly use `aria-pressed`; the All/Unread/Reading/Read chips on the shelf don't, and aren't grouped as a radio/tablist. Minor a11y inconsistency. *(Quick fix)*
- **Settings spacing/order.** The "Connect to Claude (MCP)" section is missing `mb-5`, so the "Reading history" card sits flush against it. Section order (Appearance â†’ Genres â†’ MCP â†’ Reading history) is also a little arbitrary. *(Quick fix)*
- **MCP `add_book` idempotency is per-shelf, not cross-shelf.** `addBookForUser` dedupes only within the target `ownership`, so chatting "add Dune, I own it" then "add Dune to wishlist" creates two rows for one title. Consider deduping on `dedupeKey` across shelves (move ownership instead of inserting). *(Half-day)*
- **No per-shelf privacy.** `getFriendShelf` returns every shelf (owned/wishlist/none) with ratings/reviews; loan logistics are correctly stripped. This is a known deferred decision â€” fine for v1, but a "what friends can see" toggle is a natural next step. *(Multi-day)*
- **Onboarding is empty-states only.** New users get good empty-state nudges, but the MCP connector, friend codes, and genre prefs are undiscovered unless they wander into Settings. A one-time, dismissible "here's what LibraLex can do" hint would surface them. *(Half-day)*
